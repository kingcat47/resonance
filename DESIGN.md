# 직장안전 — 암호 설계 문서

> 이후 모든 구현은 이 문서를 기준으로 삼는다.
> 이 문서는 시스템의 암호 설계, 신뢰 경계, 그리고 알려진 한계를 정식으로 기록한다.

---

## 1. 위협 모델

### 1-1. 적(Adversary) 목록

| 적 | 공격 목표 | 무엇을 보면 안 되는가 |
|---|---|---|
| 서버 운영자 | DB 직접 조회 | 신고 내용 원문, 신고자 이메일·연락처, 가해자 이름 원문 |
| DB 탈취 공격자 | 덤프 획득 | 위와 동일. 탈취 시점 기준 복호화 불가여야 함 |
| 악의적 허위 신고자 | 1인 다수 제출로 임계값 충족 | 단일 신고자가 임계값을 단독 충족하는 것 |

### 1-2. 보호되는 것 vs 보호 안 되는 것

| 구분 | 항목 |
|---|---|
| **보호됨** | 신고 내용(사건 서술, 발생 일시, 장소), 신고자 연락처·이메일, 가해자 이름/직책 원문 |
| **보호 안 됨 (설계상 한계)** | 접속 시각, IP 주소, 요청 빈도 등 네트워크 메타데이터 |
| **보호 안 됨 (의도적)** | 가해자 tag 해시값(매칭에 필요), 신고 건수(통계 목적), 계정의 존재 여부 |

설계 원칙: 서버가 알아도 되는 것은 "몇 명이 신고했는가"와 "태그 일치 여부"뿐이다.
신고 내용과 신고자 신원은 서버가 알아서는 안 된다.

## 2. 등장 행위자

### 신고자 (클라이언트)

- **아는 것**: 자신의 신고 내용 원문, 자신의 이메일, 가해자 식별 정보
- **모르는 것**: 다른 신고자의 존재, 임계값 충족 여부(알림만 수신), 감독기관 공개키 이외의 정보
- **역할**: 모든 암호화를 브라우저에서 수행하고 암호문만 서버에 전송

### API 서버

- **아는 것**: 암호화된 신고 덩어리, 가해자 tag 해시, Shamir 점(share), 계정의 해시된 식별자
- **모르는 것**: 신고 내용 원문, 신고자 이메일·연락처, 가해자 이름 원문, 비밀키
- **역할**: 태그 매칭 판정, 임계값 충족 시 감독기관에 알림, 로그인 시도 횟수 제한

### 감독기관 (복호화 권한자)

- **아는 것**: 자신의 비밀키, 임계값 충족 알림
- **모르는 것**: 임계값 미충족 신고의 내용
- **역할**: 임계값 충족 후 비밀키로 이중 잠금 해제 → 신고 내용 열람

## 3. 매칭 흐름 (핵심)

### 3-1. 가해자 tag 생성

신고자는 가해자 식별 정보(이름, 직책, 시설 식별자)를 직접 서버로 보내지 않는다.
클라이언트에서 다음 과정을 거쳐 tag를 만든다.

```
[클라이언트]
  입력: 시설ID + 가해자이름 + 직책
      ↓
  tag = HMAC-SHA256(입력값, 고정 도메인 키)
      또는
  tag = SHA256(정규화된 입력값)   ← 구현 시 결정
      ↓
  서버로 전송: tag (해시값만)

[서버]
  저장: tag (해시)
  모르는 것: 원본 이름, 직책
```

같은 가해자를 지목한 두 신고는 동일한 tag를 생성한다 → 서버는 원문 없이 일치를 판정할 수 있다.

#### 현재 방식의 한계 및 개선 방향

**한계 — 사전대입(dictionary) 공격에 취약:**
클라이언트가 서버 없이 단독으로 tag를 계산할 수 있으므로,
공격자가 "시설 X의 직원 명단"을 입력해 각각의 tag를 미리 계산한 뒤
DB의 tag 목록과 대조하면 누가 신고됐는지 역추적할 수 있다.
HMAC 고정 키가 노출되거나 SHA256 단순 해시를 쓰는 경우 더욱 위험하다.

**개선 방향 — OPRF (Oblivious Pseudo-Random Function):**
클라이언트가 입력값을 블라인딩해 서버에 보내고,
서버가 자신만 아는 키로 PRF를 적용한 뒤 블라인드 결과를 돌려주면,
클라이언트는 최종 tag를 얻지만 서버는 입력 원문을 모른다.
이렇게 하면 클라이언트 단독으로 tag를 생성할 수 없어 사전대입 공격이 차단된다.

**✓ 구현 완료 — `src/lib/crypto/oprf.ts`**
Ristretto255 기반 RFC 9497 VOPRF. 클라이언트가 블라인딩 → `POST /oprf` → 언블라인딩 → K 유도.
클라이언트 단독으로 tag를 계산할 수 없어 사전대입 공격이 차단된다.

### 3-2. Shamir 비밀분산으로 임계값 2 구현

#### 핵심 개념: 분산의 단위는 "신고자 간"이지 "한 신고 내부"가 아니다

Shamir (2, n) 비밀분산은 1차 다항식(직선) y = ax + b를 사용한다.
- **b (y절편)** = 비밀키 K (서버도 감독기관도 단독으로 모름)
- **점(share)** = 직선 위의 좌표 (x, y). x는 신고자마다 고유한 값.

같은 가해자를 지목한 신고자들이 각자 이 직선 위의 점을 1개씩 서버에 제출한다.
**점 1개 = 직선 복원 불가 = K 복원 불가.**
**서로 다른 점 2개 = 직선 복원 가능 = K 복원 가능 (임계값 충족).**

> ⚠ 이전 설계의 오류: "한 신고가 점 2개(A, B)를 만들어 하나는 감독기관에,
> 하나는 서버에 보낸다"는 구조는 신고 1건만으로도 점 2개가 모여
> 임계값이 즉시 충족되므로 근본적으로 잘못됐다. 아래가 올바른 구조다.

#### 올바른 흐름

```
[사전 준비 — 서버 또는 감독기관]

  tag마다 하나의 비밀 직선 f(x) = ax + K 가 존재한다.
    - K = 이 가해자의 신고 내용을 잠그는 대칭키 (아직 아무도 모름)
    - a = 랜덤 기울기
  직선 자체는 임계값이 충족될 때까지 어디에도 저장되지 않는다.
  (구현 방식은 1순위 작업에서 확정)


[신고자 A가 신고 제출 — 클라이언트]

  1. 신고 내용(IncidentData)을 감독기관 공개키로 암호화 → 암호문 C_A
  2. 신고자 A 고유의 x 값을 결정 (예: 계정 식별자 기반 파생)
  3. 직선 위의 점: share_A = (x_A, f(x_A))  ← 점 1개, K 복원 불가
  4. 서버로 전송:
       { tag, C_A, share_A, 신고자키로_암호화된_연락처_A }

[서버가 저장하는 것 — 신고 1건 수신 후]
  - tag (해시)
  - C_A (암호화된 신고 내용) — 서버가 못 읽음
  - share_A = (x_A, y_A) — 점 1개, K 복원 불가
  - 신고자키로_암호화된_연락처_A — 서버가 못 읽음


[신고자 B가 같은 가해자를 신고 — 클라이언트]

  1. 신고 내용을 감독기관 공개키로 암호화 → 암호문 C_B
  2. 신고자 B 고유의 x 값 결정 (x_B ≠ x_A)
  3. 직선 위의 점: share_B = (x_B, f(x_B))
  4. 서버로 전송:
       { tag, C_B, share_B, 신고자키로_암호화된_연락처_B }

[서버 — 신고 2건 수신 후]
  - tag 일치 감지 → share_A, share_B 두 점 보유
  - 라그랑주 보간으로 직선 복원 가능 → K = f(0) 계산 가능
  - 그러나 서버는 K를 직접 사용하지 않는다
  - 감독기관에 "임계값 충족" 알림 + share_A, share_B 전달
```

### 3-3. 임계값 충족 후 — 감독기관 복호화

```
[감독기관]

  share_A, share_B 수신
      ↓
  라그랑주 보간 → K = f(0) 복원
      ↓
  K는 대칭키이지만, 신고 내용 C_A / C_B는
  브라우저에서 Shamir 보간으로 K 복원
      ↓
  RSA-OAEP 비밀키로 encryptedK 언래핑 → bundle = randomKey XOR K
      ↓
  XOR 해제: randomKey = bundle XOR K
      ↓
  AES-256-GCM(randomKey)으로 C_A, C_B 복호화 → 신고 내용 원문 열람

[이중 잠금 구조]
  잠금 1 (Shamir): 같은 가해자를 지목한 서로 다른 신고자 2명의 점이 있어야
                   직선(K)을 복원할 수 있다. 1인 단독 불가.
  잠금 2 (RSA+XOR): bundle = AES임시키 XOR K 를 RSA-OAEP로 래핑.
                    RSA 비밀키만 있으면 bundle을 알지만 K 없이는 임시키 모름.
                    K만 있으면 bundle 없이는 임시키 모름.
                    둘 다 있어야 AES 복호화 가능.
  두 잠금이 모두 충족돼야 신고 내용이 열린다.
  서버는 share와 암호문을 보관하지만 단독으로 K를 복원하거나 내용을 읽을 수 없다.
```

### 3-4. 단계별 서버 가시성 요약

| 단계 | 서버가 저장하는 것 | 서버가 보는 것 | 서버가 못 보는 것 |
|---|---|---|---|
| 신고 1건 수신 | tag, C_A(암호문), share_A(점 1개), 암호화된 연락처 | tag 해시, 신고 건수 | 내용 원문, 신고자 신원, 가해자 이름, K |
| 신고 2건째 수신 | + C_B, share_B | tag 일치 → 점 2개 보유 | K (보관은 하지만 복호화 안 함) |
| 임계값 판정 | — | 점 2개로 직선 복원 가능함을 앎 | 신고 내용 (복원 안 함) |
| 감독기관 알림 | — | 알림 전송 대상 | 내용, K |
| 복호화 | — | 아무것도 없음 | 내용 (감독기관이 비밀키로 단독 복호화) |

## 4. 계정 — 제거 결정과 사유

### 결정: 계정 시스템 없음

공명 프로젝트는 계정(가입/로그인) 기능을 구현하지 않는다.
이 결정은 보안 한계 때문이 아니라 **전제 조건의 부재** 때문이다.

### 제거 사유

계정 시스템이 "1인 다수 신고로 임계값 단독 충족" 공격을 막으려면,
계정 하나 = 실제 인물 하나임이 보장되어야 한다.

그런데 이 보장은 계정 코드 자체로는 만들 수 없다:

| 방식 | 문제점 |
|---|---|
| 이메일 인증 | 임시 메일 서비스로 무제한 계정 생성 가능 |
| 전화번호 인증 | 가상 번호 서비스로 우회 가능 |
| 이메일 해시(HMAC-SHA256) | 같은 이유 — 이메일 발급 제약이 없으면 해시도 소용없음 |

결론: **이메일/전화번호 발급 자체를 외부 기관이 통제하지 않는 한**,
어떤 계정 방식도 1인 다수 계정 생성을 막지 못한다.

프라이버시를 보존하는 더 강력한 방식(예: OPAQUE 프로토콜, 익명 자격증명)도
동일한 문제에 봉착한다 — 익명성을 유지하면서 동시에 "한 사람 = 한 계정"을 보장하는 것은
**외부의 신원 검증 인프라 없이는 불가능**하다.

### 이 프로젝트의 포지셔닝

공명은 **신원 검증을 전제로 한 프라이버시 보존 반복학대 탐지 프로토타입**이다.

"신원 검증"은 이 시스템의 외부에서 이루어진다고 가정한다:
- 예: 시설 직원 인증, 기관 발급 계정, 물리적 신원 확인
- 이 시스템은 그 검증 결과를 전제로, 암호학적 프라이버시 보존 레이어를 제공한다.

신원 검증 인프라 없이 배포할 경우 1인 다수 신고 공격은 탐지되지 않는다.
이는 **알려진 한계이며, 문서화된 가정**이다.

## 5. 신뢰 경계(Trust Boundary)와 한계

### 5-1. 시스템이 보장하는 것

| 보장 항목 | 메커니즘 |
|---|---|
| 서버가 신고 내용 원문을 볼 수 없다 | AES-256-GCM + RSA-OAEP 이중 암호화. 서버는 암호문만 저장. |
| 서버가 신고자 연락처를 볼 수 없다 | 동일. 연락처도 감독기관 공개키로 암호화된 채 저장. |
| 신고 1건으로는 임계값을 충족할 수 없다 | Shamir (2,n): 점 1개로는 직선(K) 복원 불가. 수학적 보장. |
| 감독기관 비밀키 없이는 내용을 열 수 없다 | 이중 잠금: Shamir 임계값 충족 + 감독기관 RSA 비밀키 모두 필요. |

### 5-2. 시스템이 보장하지 않는 것 (명시적 한계)

| 한계 항목 | 사유 |
|---|---|
| **1인 다수 신고 방지** | 외부 신원 검증 인프라 없이는 "한 사람 = 하나의 신고자"를 보장할 수 없다. |
| 네트워크 메타데이터 보호 | 접속 IP, 시각, 요청 빈도는 서버가 관찰 가능. Tor 등 별도 수단 필요. |

### 5-3. 외부 신원 검증 전제의 의미

이 시스템을 실제 배포하려면 다음 중 하나가 선행되어야 한다:

1. **기관 계정 통합**: 이미 신원이 확인된 직원 계정(기관 LDAP, SSO 등)과 연동
2. **물리적 가입 절차**: 신고자가 오프라인에서 신원 확인 후 익명 토큰 발급
3. **정부/기관 발급 자격증명**: 예: 공무원 인증서, 기관 이메일 도메인 제한

위 전제 없이 배포하면 "공격자가 임계값을 단독 충족해 허위 신고를 유도"하는 공격이 가능하다.
이 공격은 시스템 내부 코드로는 막을 수 없다.

## 6. 조회 / 재접속 토큰

### 현재 구조: 1회 표시 토큰

신고 제출 완료 시 `XXXX-XXXX-XXXX` 형식의 재접속 토큰을 1회 표시한다.
토큰은 메모리에만 존재하고 서버/스토리지에 저장되지 않는다.

계정 시스템이 없으므로 이 토큰이 신고자가 자신의 신고를 재확인하는 유일한 방법이다.
의도적 설계: 토큰을 잃으면 재접속 불가. 이는 "서버가 신고자를 추적할 수 없다"는 원칙과 일치한다.

### 향후 개선 방향

외부 신원 검증 인프라가 연동되는 경우, 계정 기반 조회로 교체 가능하다.
그 전까지는 1회 표시 토큰이 공식 방식이다.

## 7. 구현 우선순위

### 1순위 — 매칭 코어 (Shamir) 직접 구현 + 유닛테스트 ✓ 완료

**라이브러리 방침:**
- Shamir 프로토콜 로직(1차 다항식 생성, 라그랑주 보간, 유한체 GF(p) 위의 연산)은 직접 구현한다.
  프로토콜 구조를 블랙박스 라이브러리에 숨기면 설계 검증이 불가능하기 때문이다.
- 단, 해시(SHA-256, HMAC), 난수 생성, AES-GCM 등 암호 프리미티브는
  **Web Crypto API** 또는 검증된 라이브러리(예: @noble/hashes, @noble/curves)를 사용한다.

**구현 완료:**
- `src/lib/crypto/shamir.ts`: GF(secp256k1 소수) 위 Shamir (2,n) — deriveLine, makeShare, recoverK
- `src/lib/crypto/tag.ts`: HMAC-SHA256 기반 tag 생성
- 유닛테스트 9개 통과 (shamir.test.ts)

### 2순위 — 실제 암호 교체 ✓ 완료

**구현 완료:**
- `src/lib/crypto/encrypt.ts`: AES-256-GCM + RSA-2048-OAEP 하이브리드 암호화
- 유닛테스트 5개 통과 (encrypt.test.ts)
- 신고 페이지(Report)에 실제 암호 함수 연결 완료

### 3순위 — 백엔드 ✓ 완료

**구현 완료:**
- NestJS 10 + Prisma 6 + PostgreSQL 16 (Docker)
- `POST /reports`: 신고 수신 및 tag 매칭 판정
- `GET /reports/:tag/status`: 매칭 상태 조회
- 유닛테스트 5개 통과 (reports.service.spec.ts)

### 4순위 — 감독기관 콘솔 ✓ 완료

**구현 완료:**
- `/admin` 페이지: 감독기관 전용 콘솔 (헤더 없음, 독립 라우트)
- `GET /admin/matches`: `x-admin-key` 헤더 인증, 임계값 충족 tag 목록 + 암호문 반환
- 브라우저에서 RSA 비밀키 JWK 붙여넣기 → Shamir 보간(K 복원) + AES-256-GCM 복호화
- 증거 파일 이미지 미리보기 및 다운로드 지원
- 비밀키는 브라우저 메모리에만 존재, 서버로 전송되지 않음
